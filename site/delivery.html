Why the governance frame holds— the load-bearing design decisions
01

Federated identity, zero secrets

Identity is OIDC-federated — no client secrets anywhere in CI/CD, from Challenge 00 onward.

02

One validator, two places

The same validate code path runs on a laptop and in CI — no surprises between local and pipeline.

03

Drift runs nightly

Out-of-band portal changes surface automatically as GitHub issues — provable "no ClickOps".

04

Skills do the heavy lifting

Skills for Fabric ship medallion, Eventhouse, Power BI, and Activator work — wrapped in the PR loop so the frame still holds.

05

Agents accelerate, never bypass

GitHub Copilot and Fabric MCP speed authoring and review. Every change is still a reviewed PR applied by a service principal.

06

Each delivery feeds the next

Customer feedback becomes PRs that add or adapt challenges, so the blueprint compounds value over time.

Evolve the blueprint— feedback maps to one of four PR types
fix:

Bug or doc rot in an existing challenge — a broken link in 03, an outdated screenshot in 07.

enhance:

Sharpen an existing challenge — a new policy rule in 01, clearer success criteria in 04.

adapt:

A customer-specific variant kept alongside the original — a regulated-insurance take on 06.

feat:

A brand-new challenge — e.g. data-mesh federated governance for multi-domain tenants.

Every PR that adds or changes challenge content keeps the contract: each challenge demonstrates ≥ 1 MCP server and ≥ 1 Skill, and policy / schema changes need CODEOWNERS review.
